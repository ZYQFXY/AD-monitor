# 跨境广告 AI 监控项目 · 第二阶段规划

**前提条件：** 第一阶段稳定运行 4~8 周，规则预警准确率 ≥ 85%，adset_snapshots / ad_snapshots 积累足量历史数据。

---

## 一、整体路线

```
第一阶段（已完成）
  ↓ 4~8 周数据积累 + 准确率评估
阶段 2.1  AI 给建议 → 投手企微卡片一键确认执行
  ↓ 2 周仿真期评测，采纳率 ≥ 70%
阶段 2.2  小额低风险操作允许 AI 自动执行（±10% 调价），超阈值仍需人工确认
  ↓ 持续评估 ROI 影响，逐步扩大授权范围
阶段 2.3  按投手分级授权，开放更多自动决策范围
```

---

## 二、各阶段功能详细说明

### 2.1 AI 建议 + 人工一键确认

**目标：** AI 替代投手"判断该怎么做"，投手只需点"确认/忽略"，减少决策时间。

**新增功能：**

| 功能 | 说明 |
|------|------|
| AI 决策引擎 | 输入 AdSet 历史趋势 → Claude API → 输出操作建议（关停/加预算/减预算） |
| 企微交互卡片 | 发送带按钮的消息卡片（确认执行 / 忽略），投手点击后回调系统 |
| 操作回调接口 | 接收企微卡片按钮回调，记录决策结果（采纳/忽略） |
| 操作记录表 | 记录每条 AI 建议的状态：pending / accepted / rejected |
| 人工操作日志 | 投手在平台执行操作后，可在 Web 界面手动记录结果（ROI 变化），用于评测 |

**企微卡片样例：**
```
[AI 决策建议]
平台：Facebook | 品类：女装
广告组：广告组-A（fb_adset_001）
当前 ROAS：0.85（连续 3 天下滑）
近 7 天消耗趋势：↑ +23%
AI 建议：暂停该广告组
置信度：82%

[确认执行] [暂不处理]
```

---

### 2.2 小额自动执行 + 超阈值人工确认

**目标：** 高置信度的低风险操作（预算微调）由 AI 自动执行，节省投手操作时间。

**新增功能：**

| 功能 | 说明 |
|------|------|
| 平台写接口 | 接入三平台 API 写接口：暂停 AdSet、调整日预算 |
| 自动执行引擎 | 满足条件时直接调用写接口，不等待人工确认 |
| 执行风控规则 | 单次调价幅度 ≤ 10%；日累计消耗削减 ≤ 20%；不允许 AI 自动删除广告 |
| 操作通知 | 自动执行后发企微通知，告知投手已执行的操作及理由 |
| 一键撤销 | 企微通知中提供"撤销"按钮，投手可在 30 分钟内回滚操作 |

**自动执行决策树：**
```
AI 建议操作
  ├── 操作类型 = 暂停 AdSet
  │     ├── 日消耗 < 100 元 → 自动执行
  │     └── 日消耗 ≥ 100 元 → 发卡片，人工确认
  ├── 操作类型 = 增加预算
  │     ├── 调整幅度 ≤ 10% → 自动执行
  │     └── 调整幅度 > 10% → 人工确认
  └── 操作类型 = 关停 Campaign → 始终人工确认
```

---

### 2.3 分级授权 + 扩大自动决策范围

**目标：** 基于历史采纳率和投手信任度，动态扩大 AI 自动执行权限。

**新增功能：**

| 功能 | 说明 |
|------|------|
| 投手信任度评分 | 基于过去 N 周 AI 建议采纳率、执行后 ROI 变化，自动计算信任度分数 |
| 分级授权配置 | Web 界面为每个投手/品类配置 AI 操作权限级别（L1/L2/L3） |
| 扩展操作范围 | L3 级别可授权 AI 自动执行：素材暂停、预算 ±20%、跨 AdSet 预算迁移 |
| 异常熔断 | 检测到 ROI 连续下滑或 AI 操作引发异常消耗，自动降级为人工确认模式 |

### 3.2 数据库新增表

| 表名 | 用途 |
|------|------|
| `ai_suggestions` | AI 建议记录：adset_id、action_type、reason、confidence、status（pending/accepted/rejected/auto_executed）、created_at |

### 3.3 整体架构流程

```

        │
        ▼

  · 近7/14/30天ROAS趋势斜率

        │

        │                               │
        │                         投手点击"确认"
        │                               │
        └── 高置信度 / 低风险 ──→ 操作执行器（自动执行）
                                        │
                               三平台写接口执行
                               action_records 入库
                               企微通知"已执行 + 撤销入口"
```

## 四、AI 决策引擎实现

### 4.1 特征提取（features.py）

从 `adset_snapshots` 表提取以下特征供 Claude 分析：

| 特征 | 计算方式 |
|------|----------|
| ROAS 趋势 | 近 7/14/30 天 ROAS 均值及斜率（线性回归） |
| 消耗加速度 | 今日消耗 / 近 7 天日均消耗 |
| 预算利用率 | spend / daily_budget |
| 告警频率 | 近 30 天同规则触发次数 |
| 历史采纳率 | 该 AdSet 历史 AI 建议被采纳的比例 |

### 4.2 调用 Claude API（decision.py）

使用 **claude-opus-4-7** 或 **claude-sonnet-4-6**（成本敏感时）作为决策模型。

**Prompt 结构：**

```
系统提示（System）：
  你是一个跨境广告优化助手，根据广告组的历史表现数据，
  给出暂停/增预算/减预算的建议，并说明理由。
  输出格式：JSON { action, confidence, reason, risk_level }

用户消息（User）：
  品类：{category_name}
  平台：{platform}
  广告组：{adset_name}（{adset_id}）
  近7天ROAS：{roas_7d}（趋势：{trend}）
  今日消耗：¥{spend}（加速度：{acceleration}）
  预算利用率：{budget_usage}
  投手阈值配置：min_roas={min_roas}, scale_roas={scale_roas}
  历史操作参考：{past_actions}

  请给出操作建议。
```

**输出示例：**
```json
{
  "action": "pause",
  "confidence": 0.87,
  "reason": "ROAS 连续 4 天低于红线，消耗加速 1.3 倍，历史相似场景中暂停后 3 天 ROAS 回升",
  "risk_level": "low"
}
```

### 4.3 决策分发逻辑

```python
if suggestion["confidence"] >= 0.85 and suggestion["risk_level"] == "low":
    if action_within_auto_limit(suggestion, pitcher_trust_level):
        executor.auto_execute(suggestion)   # 自动执行（2.2 阶段开放）
    else:
        wecom.send_confirm_card(suggestion)  # 发交互卡片
else:
    wecom.send_confirm_card(suggestion)      # 置信度不够，发卡片
```

## 六、三平台写接口实现

> 第二阶段开始调用**写接口**，风险等级提升。所有写操作必须记录到 `action_records` 表，且支持回滚。

### 6.1 各平台支持的操作

| 操作 | Facebook | Google Ads | TikTok |
|------|----------|------------|--------|
| 暂停 AdSet | `AdSet.remote_update(status=PAUSED)` | `MutateAdGroups(status=PAUSED)` | `UPDATE /adgroup/status/` |
| 恢复 AdSet | `status=ACTIVE` | `status=ENABLED` | `status=ENABLE` |
| 调整日预算 | `AdSet.remote_update(daily_budget=新值)` | `MutateCampaignBudgets(amount_micros=新值)` | `UPDATE /adgroup/budget/` |

### 6.2 执行器设计（executor.py）

```python
def execute(suggestion: dict, executor_type: str = "human") -> dict:
    platform = suggestion["platform"]
    action = suggestion["action"]

    # 写操作前记录
    record_id = create_action_record(suggestion["id"], executor_type)

    try:
        if platform == "facebook":
            result = facebook_write.execute(action, suggestion)
        elif platform == "google":
            result = google_write.execute(action, suggestion)
        elif platform == "tiktok":
            result = tiktok_write.execute(action, suggestion)

        update_action_record(record_id, status="success", rollback_token=result.get("rollback_token"))
        return {"success": True, "record_id": record_id}

    except Exception as e:
        update_action_record(record_id, status="failed", error=str(e))
        return {"success": False, "error": str(e)}
```

### 6.3 回滚机制

- 暂停操作：`rollback_token = adset_id`，调用"恢复"接口即可回滚
- 预算调整：`rollback_token = 原预算值`，恢复原值即可回滚
- 回滚入口：企微通知中的"撤销"链接有效期 **30 分钟**

## 七、评测体系

### 7.1 上线 AI 建议前（2.1 阶段前置）

| 评测项 | 方法 | 通过标准 |
|--------|------|----------|
| 回测准确率 | 用第一阶段历史数据，AI 建议与投手实际操作对比 | 一致率 ≥ 85% |
| 仿真期 | AI 建议静默运行 2 周，不实际执行，记录与人工决策的差异 | 采纳率 ≥ 70% |
| ROI 影响 | 抽样对比：采纳 AI 建议 vs 未采纳，3 天后 ROAS 变化 | 采纳后 ROAS ≥ 人工操作均值 |

### 7.2 上线后持续监控

| 指标 | 计算方式 | 预警阈值 |
|------|----------|----------|
| 建议采纳率 | accepted / (accepted + rejected) | 连续 2 周 < 50% → 降级 |
| 自动执行成功率 | success / total_auto_executed | < 95% → 暂停自动执行 |
| AI 执行后 ROI 变化 | 执行后 3 天 ROAS vs 执行前均值 | 下降 > 15% → 触发熔断 |
| 回滚率 | rollback / total_executed | > 20% → 降级为人工确认 |

### 7.3 熔断机制

满足以下任一条件，自动降级为全人工确认模式：
- AI 执行后 ROI 连续 3 天下滑
- 回滚率超过 20%
- 单日自动执行失败次数 ≥ 3
- 投手手动触发"暂停 AI 自动执行"

## 十一、模型准确率不足时的改进路径

### 11.1 准确率不足的判断标准

满足以下任一条件，认为当前模型准确率不足，需要介入改进：

| 指标 | 不足阈值 | 说明 |
|------|----------|------|
| 建议采纳率 | < 60% | 投手频繁忽略 AI 建议，说明建议与实际判断偏差大 |
| 采纳后 ROI 下滑 | ROAS 下降 > 10% | AI 建议被采纳后效果反而变差 |
| 回测一致率 | < 75% | 历史数据回测时，AI 与投手操作差异过大 |
| 投手主观反馈 | 建议"太激进"或"太保守" | 说明模型对该品类/平台理解偏差 |

### 11.2 改进优先级（由易到难，逐步尝试）

```
Level 1  优化 Prompt（零成本，立即可做）
    ↓    若仍不足
Level 2  RAG 注入历史决策（中等成本，2 周）
    ↓    若仍不足
Level 3  Fine-tune 模型（高成本，需积累数据）
    ↓    若仍不足
Level 4  自训练专用 ML 模型（最高成本，备用方案）
```

**原则：不要跳级。Fine-tune 之前必须穷尽 Prompt 优化和 RAG。**

---
### 11.3 Level 1：优化 Prompt（零成本，优先尝试）

**适用场景：** 建议方向大致正确，但阈值判断偏激进或偏保守。

**优化方向：**

| 问题现象 | Prompt 调整方法 |
|----------|----------------|
| 建议太激进（频繁建议暂停） | 在 System Prompt 加入"谨慎原则"，要求 ROAS 连续 N 天低于红线才建议暂停 |
| 建议太保守（高消耗低ROAS 不敢建议关停） | 注入该品类历史上投手主动关停时的典型数据作为 few-shot 示例 |
| 忽视平台差异 | 在 Prompt 中说明各平台数据延迟特性（FB/Google 延迟1~3h，TikTok 1h），调整判断时间窗口 |
| 建议理由模糊 | 要求输出 chain-of-thought（先分析再结论），提升可解释性和准确性 |

**Few-shot 示例格式（注入 Prompt）：**
```
以下是投手过去的真实操作记录，请参考判断风格：

案例1：
  广告组：fb_adset_xxx | ROAS近7天：[1.5, 1.3, 1.1, 0.9, 0.85, 0.8, 0.75]（持续下滑）
  今日消耗：¥1200 | 投手操作：暂停
  → 结论：ROAS连续7天下滑且低于红线，消耗大，投手选择暂停

案例2：
  广告组：gg_adset_yyy | ROAS近7天：[2.8, 3.0, 3.2, 3.5, 3.3, 3.6, 3.8]（持续上升）
  今日消耗：¥600（日预算¥2000未达上限）| 投手操作：增加预算+20%
  → 结论：ROAS高且上升，预算有余量，投手选择放量
```

### 11.4 Level 2：RAG 注入历史决策（中等成本）

**适用场景：** Prompt 优化后准确率仍低于目标，需要让模型"记住"该团队的决策风格。

**核心思路：** 每次做决策前，从历史 `action_records` 中检索最相似的案例，注入 Prompt 作为上下文。

**实现步骤：**

```
Step 1  向量化历史决策
        将每条 action_records 的特征（ROAS趋势、消耗、平台、品类）
        转换为向量，存入向量数据库（推荐 ChromaDB，本地部署，免费）

Step 2  检索相似案例
        新 AdSet 决策时，计算当前特征向量与历史案例的相似度
        取 Top-3 最相似且被投手采纳的案例

Step 3  注入 Prompt
        将检索到的案例作为 few-shot 示例注入 Claude API 调用
        "以下是该团队过去在类似情况下的真实操作..."

Step 4  效果验证
        对比 RAG 前后的采纳率变化，通常可提升 10~20%
```

**新增依赖：**
```
chromadb==0.5.0        # 本地向量数据库
sentence-transformers  # 特征向量化（或直接用 Claude Embeddings API）
```

**新增文件：**
```
ai/
├── rag.py             # 向量化 + 检索逻辑
└── vector_store/      # ChromaDB 本地存储目录
```

### 11.5 Level 3：Fine-tune 模型

**适用场景：** RAG 后准确率仍不足，且已积累 **500+ 条**投手决策记录（约运营 6 个月后）。

**数据量要求：**

| 训练数据量 | 预期效果 |
|-----------|----------|
| < 200 条 | 不建议 Fine-tune，数据太少容易过拟合 |
| 200~500 条 | 可做初步 Fine-tune，效果有限 |
| 500~2000 条 | Fine-tune 效果明显，推荐启动 |
| > 2000 条 | 效果稳定，可考虑区分品类/平台独立模型 |

**Fine-tune 流程（以 Anthropic Fine-tuning API 为例）：**

```
Step 1  准备训练数据（JSONL 格式）
        从 action_records + adset_snapshots 提取被投手采纳的决策
        每条格式：
        {
          "messages": [
            {"role": "system", "content": "你是广告优化助手..."},
            {"role": "user", "content": "ROAS近7天趋势：...今日消耗：..."},
            {"role": "assistant", "content": "{\"action\":\"pause\",\"confidence\":0.9,\"reason\":\"...\"}"}
          ]
        }

Step 2  数据清洗
        只保留投手采纳的决策（accepted）
        剔除异常数据（ROI 异常波动期）
        按 action_type 均衡采样，避免"暂停"类别占比过高

Step 3  拆分数据集
        训练集：80%  验证集：10%  测试集：10%

Step 4  提交 Fine-tune 任务
        调用 Anthropic Fine-tuning API
        选择基础模型：claude-haiku-4-5（成本最低）

Step 5  评测 Fine-tuned 模型
        在测试集上对比 Fine-tuned 模型 vs 原始 Claude API
        指标：action_type 分类准确率、置信度校准

Step 6  A/B 上线
        Fine-tuned 模型处理 20% 流量，原模型处理 80%
        对比采纳率和 ROI 影响，稳定后全量切换
```

**注意：** Anthropic 目前公开支持 Fine-tune 的模型以 Haiku 系列为主，
Opus/Sonnet 的 Fine-tune 需联系 Anthropic 商务，费用更高。

### 11.6 Level 4：自训练专用 ML 模型（备用方案）

**适用场景：** Fine-tune 成本过高，或需要实时推理（不依赖外部 API）。

**模型选型：**

对于广告投放决策这类**结构化表格数据**，传统 ML 模型通常比 LLM 效率更高：

| 模型 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| XGBoost / LightGBM | 表格数据效果好，训练快，可解释性强 | 需要足够数量标注数据 | ★★★★★ |
| 随机森林 | 简单稳健，不易过拟合 | 准确率略低于 XGBoost | ★★★★ |
| 神经网络（MLP） | 可捕捉非线性关系 | 数据量少时容易过拟合 | ★★★ |

**推荐方案：XGBoost 多分类模型**

**完整训练流程：**

```
Step 1  特征工程
        从 adset_snapshots 计算特征向量（每个 AdSet 一行）：
        · roas_7d_mean    近7天ROAS均值
        · roas_7d_slope   近7天ROAS趋势斜率（线性回归系数）
        · spend_accel     今日消耗 / 近7天日均消耗
        · budget_usage    spend / daily_budget
        · ctr_7d_mean     近7天CTR均值
        · alert_count_30d 近30天触发预警次数
        · days_since_create AdSet 上线天数
        · platform        one-hot 编码（fb/google/tiktok）

Step 2  标注标签
        从 action_records 中取投手操作：
        0 = no_action（无需操作）
        1 = pause（暂停）
        2 = increase_budget（增加预算）
        3 = decrease_budget（减少预算）

Step 3  数据预处理
        train_test_split(test_size=0.2, stratify=y)
        处理类别不平衡（no_action 通常占多数）：SMOTE 过采样或 class_weight

Step 4  模型训练
        import xgboost as xgb
        model = xgb.XGBClassifier(
            n_estimators=200,
            max_depth=6,
            learning_rate=0.05,
            use_label_encoder=False,
            eval_metric='mlogloss'
        )
        model.fit(X_train, y_train,
                  eval_set=[(X_val, y_val)],
                  early_stopping_rounds=20)

Step 5  评估
        classification_report(y_test, y_pred)
        重点关注 pause 和 increase_budget 的 precision/recall
        目标：precision ≥ 0.80（减少误操作）

Step 6  模型保存 + 服务化
        joblib.dump(model, 'ai/models/ad_decision_v1.pkl')
        在 ai/decision.py 中加载并调用，替代 Claude API
        或与 Claude API 并行运行做集成（取置信度加权平均）

Step 7  持续训练（每月更新）
        新增一个月数据后重新训练
        对比新旧模型测试集指标，只有新模型更优才发布
```

**新增文件：**
```
ai/
├── train.py            # 训练脚本（离线运行）
├── evaluate.py         # 评估脚本
└── models/
    └── ad_decision_v1.pkl   # 训练好的模型文件
```

### 11.7 改进路径决策树

```
准确率不足（采纳率 < 60% 或 ROI 下滑）
        │
        ▼
[Level 1] 优化 Prompt + Few-shot 示例
        │ 运行 2 周，评估采纳率
        ├── 达标（≥70%）→ 继续使用 Claude API
        │
        ▼ 未达标
[Level 2] RAG 注入历史决策（ChromaDB）
        │ 运行 2 周，评估采纳率
        ├── 达标 → 继续（RAG + Claude API）
        │
        ▼ 未达标 且 已积累 500+ 条决策数据
[Level 3] Fine-tune Claude Haiku
        │ 数据准备 2 周 + 训练 + A/B 测试 3 周
        ├── 达标 → 使用 Fine-tuned 模型
        │
        ▼ Fine-tune 成本过高 / 不依赖外部 API
[Level 4] 自训练 XGBoost 模型（本地推理）
          数据准备 + 训练 + 评估 + 上线
```

### 11.8 各方案对比

| 方案 | 启动门槛 | 预计耗时 | 成本 | 准确率提升 | 适用阶段 |
|------|----------|----------|------|-----------|----------|
| Prompt 优化 | 无，立即可做 | 1~2 天 | 零额外成本 | +5~15% | 任何时候 |
| RAG | 200+ 条历史决策 | 2 周 | 低（ChromaDB免费） | +10~20% | 运营 2~3 月后 |
| Fine-tune | 500+ 条被采纳决策 | 4~6 周 | 中（API调用费） | +15~30% | 运营 6 月后 |
| 自训练 ML | 1000+ 条标注数据 | 6~8 周 | 低（本地训练） | +10~25% | 运营 6~12 月后 |

### 11.9 数据积累策略（为改进做准备）

从第一阶段开始就要为将来的模型训练做数据准备：

1. **每条告警都记录投手的实际操作**
   - 在 Web 界面告警历史页增加"投手反馈"字段（已操作/忽略/操作但无效）
   - 这是 Fine-tune 和 ML 训练的核心标注数据

2. **记录操作后的效果**
   - 操作后 1/3/7 天的 ROAS 变化自动计算并回填到 `action_records`
   - 用于评估操作质量，是 RAG 检索时的权重依据

3. **避免数据污染**
   - 标注数据只取**投手主动操作**的记录，AI 自动执行的记录单独存储
   - 防止"AI 训练 AI"的正反馈循环
