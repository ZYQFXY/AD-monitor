# 跨境广告 AI 监控项目 · 第二阶段规划

**前提条件：** 第一阶段稳定运行 4~8 周，规则预警准确率 ≥ 85%，adset_snapshots / ad_snapshots 积累足量历史数据。

---

## 一、整体路线

```
第一阶段（已完成）
  ↓ 4~8 周数据积累 + 准确率评估
阶段 2.1  AI 给建议 → 投手企微卡片一键确认执行
  ↓ 2 周仿真期评测，采纳率 ≥ 70%
阶段 2.2  小额低风险操作允许 AI 自动执行（±10% 调价），超阈值仍需人工确认
  ↓ 持续评估 ROI 影响，逐步扩大授权范围
阶段 2.3  按投手分级授权，开放更多自动决策范围
```

---

## 二、各阶段功能详细说明

### 2.1 AI 建议 + 人工一键确认

**目标：** AI 替代投手"判断该怎么做"，投手只需点"确认/忽略"，减少决策时间。

**新增功能：**

| 功能 | 说明 |
|------|------|
| AI 决策引擎 | 输入 AdSet 历史趋势 → Claude API → 输出操作建议（关停/加预算/减预算） |
| 企微交互卡片 | 发送带按钮的消息卡片（确认执行 / 忽略），投手点击后回调系统 |
| 操作回调接口 | 接收企微卡片按钮回调，记录决策结果（采纳/忽略） |
| 操作记录表 | 记录每条 AI 建议的状态：pending / accepted / rejected |
| 人工操作日志 | 投手在平台执行操作后，可在 Web 界面手动记录结果（ROI 变化），用于评测 |

**企微卡片样例：**
```
[AI 决策建议]
平台：Facebook | 品类：女装
广告组：广告组-A（fb_adset_001）
当前 ROAS：0.85（连续 3 天下滑）
近 7 天消耗趋势：↑ +23%
AI 建议：暂停该广告组
置信度：82%

[确认执行] [暂不处理]
```

---

### 2.2 小额自动执行 + 超阈值人工确认

**目标：** 高置信度的低风险操作（预算微调）由 AI 自动执行，节省投手操作时间。

**新增功能：**

| 功能 | 说明 |
|------|------|
| 平台写接口 | 接入三平台 API 写接口：暂停 AdSet、调整日预算 |
| 自动执行引擎 | 满足条件时直接调用写接口，不等待人工确认 |
| 执行风控规则 | 单次调价幅度 ≤ 10%；日累计消耗削减 ≤ 20%；不允许 AI 自动删除广告 |
| 操作通知 | 自动执行后发企微通知，告知投手已执行的操作及理由 |
| 一键撤销 | 企微通知中提供"撤销"按钮，投手可在 30 分钟内回滚操作 |

**自动执行决策树：**
```
AI 建议操作
  ├── 操作类型 = 暂停 AdSet
  │     ├── 日消耗 < 100 元 → 自动执行
  │     └── 日消耗 ≥ 100 元 → 发卡片，人工确认
  ├── 操作类型 = 增加预算
  │     ├── 调整幅度 ≤ 10% → 自动执行
  │     └── 调整幅度 > 10% → 人工确认
  └── 操作类型 = 关停 Campaign → 始终人工确认
```

---

### 2.3 分级授权 + 扩大自动决策范围

**目标：** 基于历史采纳率和投手信任度，动态扩大 AI 自动执行权限。

**新增功能：**

| 功能 | 说明 |
|------|------|
| 投手信任度评分 | 基于过去 N 周 AI 建议采纳率、执行后 ROI 变化，自动计算信任度分数 |
| 分级授权配置 | Web 界面为每个投手/品类配置 AI 操作权限级别（L1/L2/L3） |
| 扩展操作范围 | L3 级别可授权 AI 自动执行：素材暂停、预算 ±20%、跨 AdSet 预算迁移 |
| 异常熔断 | 检测到 ROI 连续下滑或 AI 操作引发异常消耗，自动降级为人工确认模式 |

## 三、技术架构变化

### 3.1 新增模块

```
ad-monitor/
├── ai/
│   ├── decision.py          # AI 决策引擎（调用 Claude API）
│   └── features.py          # 特征提取（从快照表计算历史趋势）
│
├── actions/
│   ├── executor.py          # 操作执行器（调用三平台写接口）
│   ├── facebook_write.py    # Facebook 暂停/调预算
│   ├── google_write.py      # Google Ads 暂停/调预算
│   └── tiktok_write.py      # TikTok 暂停/调预算
│
└── web/templates/
    └── ai_decisions.html    # AI 建议历史 + 执行状态页
```

### 3.2 数据库新增表

| 表名 | 用途 |
|------|------|
| `ai_suggestions` | AI 建议记录：adset_id、action_type、reason、confidence、status（pending/accepted/rejected/auto_executed）、created_at |
| `action_records` | 实际执行记录：suggestion_id、executed_at、executor（human/ai）、result（success/failed）、rollback_token |
| `pitcher_trust` | 投手信任度：pitcher_userid、category_id、acceptance_rate、roi_impact、trust_level（L1/L2/L3） |

### 3.3 整体架构流程

```
历史快照（adset_snapshots）
        │
        ▼
  特征提取（features.py）
  · 近7/14/30天ROAS趋势斜率
  · 消耗加速度（今日/昨日/7日均值）
  · 历史人工操作时机
        │
        ▼
  Claude API（decision.py）
  输入：特征数据 + 品类阈值 + 历史决策
  输出：建议操作 + 置信度 + 理由
        │
        ├── 低置信度 / 高风险 ──→ 企微交互卡片（人工确认）
        │                               │
        │                         投手点击"确认"
        │                               │
        └── 高置信度 / 低风险 ──→ 操作执行器（自动执行）
                                        │
                               三平台写接口执行
                               action_records 入库
                               企微通知"已执行 + 撤销入口"
```

## 四、AI 决策引擎实现

### 4.1 特征提取（features.py）

从 `adset_snapshots` 表提取以下特征供 Claude 分析：

| 特征 | 计算方式 |
|------|----------|
| ROAS 趋势 | 近 7/14/30 天 ROAS 均值及斜率（线性回归） |
| 消耗加速度 | 今日消耗 / 近 7 天日均消耗 |
| 预算利用率 | spend / daily_budget |
| 告警频率 | 近 30 天同规则触发次数 |
| 历史采纳率 | 该 AdSet 历史 AI 建议被采纳的比例 |

### 4.2 调用 Claude API（decision.py）

使用 **claude-opus-4-7** 或 **claude-sonnet-4-6**（成本敏感时）作为决策模型。

**Prompt 结构：**

```
系统提示（System）：
  你是一个跨境广告优化助手，根据广告组的历史表现数据，
  给出暂停/增预算/减预算的建议，并说明理由。
  输出格式：JSON { action, confidence, reason, risk_level }

用户消息（User）：
  品类：{category_name}
  平台：{platform}
  广告组：{adset_name}（{adset_id}）
  近7天ROAS：{roas_7d}（趋势：{trend}）
  今日消耗：¥{spend}（加速度：{acceleration}）
  预算利用率：{budget_usage}
  投手阈值配置：min_roas={min_roas}, scale_roas={scale_roas}
  历史操作参考：{past_actions}

  请给出操作建议。
```

**输出示例：**
```json
{
  "action": "pause",
  "confidence": 0.87,
  "reason": "ROAS 连续 4 天低于红线，消耗加速 1.3 倍，历史相似场景中暂停后 3 天 ROAS 回升",
  "risk_level": "low"
}
```

### 4.3 决策分发逻辑

```python
if suggestion["confidence"] >= 0.85 and suggestion["risk_level"] == "low":
    if action_within_auto_limit(suggestion, pitcher_trust_level):
        executor.auto_execute(suggestion)   # 自动执行（2.2 阶段开放）
    else:
        wecom.send_confirm_card(suggestion)  # 发交互卡片
else:
    wecom.send_confirm_card(suggestion)      # 置信度不够，发卡片
```

### 5.2 Web 确认页（ai_decisions.html）

投手通过企微消息中的链接进入内网确认页，页面展示：

- AI 建议操作详情

- 「确认执行」/ 「忽略」按钮

**路由：**

## 六、三平台写接口实现

> 第二阶段开始调用**写接口**，风险等级提升。所有写操作必须记录到 `action_records` 表，且支持回滚。

### 6.1 各平台支持的操作

| 操作 | Facebook | Google Ads | TikTok |
|------|----------|------------|--------|
| 暂停 AdSet | `AdSet.remote_update(status=PAUSED)` | `MutateAdGroups(status=PAUSED)` | `UPDATE /adgroup/status/` |
| 恢复 AdSet | `status=ACTIVE` | `status=ENABLED` | `status=ENABLE` |
| 调整日预算 | `AdSet.remote_update(daily_budget=新值)` | `MutateCampaignBudgets(amount_micros=新值)` | `UPDATE /adgroup/budget/` |

### 6.2 执行器设计（executor.py）

```python
def execute(suggestion: dict, executor_type: str = "human") -> dict:
    platform = suggestion["platform"]
    action = suggestion["action"]

    # 写操作前记录
    record_id = create_action_record(suggestion["id"], executor_type)

    try:
        if platform == "facebook":
            result = facebook_write.execute(action, suggestion)
        elif platform == "google":
            result = google_write.execute(action, suggestion)
        elif platform == "tiktok":
            result = tiktok_write.execute(action, suggestion)

        update_action_record(record_id, status="success", rollback_token=result.get("rollback_token"))
        return {"success": True, "record_id": record_id}

    except Exception as e:
        update_action_record(record_id, status="failed", error=str(e))
        return {"success": False, "error": str(e)}
```

### 6.3 回滚机制

- 暂停操作：`rollback_token = adset_id`，调用"恢复"接口即可回滚
- 预算调整：`rollback_token = 原预算值`，恢复原值即可回滚
- 回滚入口：企微通知中的"撤销"链接有效期 **30 分钟**

## 七、评测体系

### 7.1 上线 AI 建议前（2.1 阶段前置）

| 评测项 | 方法 | 通过标准 |
|--------|------|----------|
| 回测准确率 | 用第一阶段历史数据，AI 建议与投手实际操作对比 | 一致率 ≥ 85% |
| 仿真期 | AI 建议静默运行 2 周，不实际执行，记录与人工决策的差异 | 采纳率 ≥ 70% |
| ROI 影响 | 抽样对比：采纳 AI 建议 vs 未采纳，3 天后 ROAS 变化 | 采纳后 ROAS ≥ 人工操作均值 |

### 7.2 上线后持续监控

| 指标 | 计算方式 | 预警阈值 |
|------|----------|----------|
| 建议采纳率 | accepted / (accepted + rejected) | 连续 2 周 < 50% → 降级 |
| 自动执行成功率 | success / total_auto_executed | < 95% → 暂停自动执行 |
| AI 执行后 ROI 变化 | 执行后 3 天 ROAS vs 执行前均值 | 下降 > 15% → 触发熔断 |
| 回滚率 | rollback / total_executed | > 20% → 降级为人工确认 |

### 7.3 熔断机制

满足以下任一条件，自动降级为全人工确认模式：
- AI 执行后 ROI 连续 3 天下滑
- 回滚率超过 20%
- 单日自动执行失败次数 ≥ 3
- 投手手动触发"暂停 AI 自动执行"

## 九、风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI 建议错误导致亏损 | 高 | 2.1 阶段纯建议不执行；2.2 仅开放低风险小额操作；熔断机制兜底 |
| 平台写接口权限不足 | 中 | 提前申请广告账户写权限（需 Facebook Business Manager 管理员审批） |
| Claude API 延迟 / 超时 | 低 | 设置 10s 超时，超时降级为纯规则预警，不影响第一阶段功能 |
| 投手误点确认 | 中 | 确认页展示操作预览 + 二次弹窗确认；执行后 30 分钟内可撤销 |
| 多投手并发操作冲突 | 低 | 同一 AdSet 同时只能有一个 pending 建议，乐观锁防并发 |
| 第一阶段数据量不足 | 中 | 每个 AdSet 至少需要 30 天快照数据才启用 AI 建议，数据不足时保持规则预警 |

## 十、总结

| 阶段 | 核心价值 | 启动条件 | 预计耗时 |
|------|----------|----------|----------|
| **2.1** AI 建议 + 人工确认 | 减少投手判断时间，AI 做建议 | 第一阶段运行 4~8 周，准确率 ≥ 85% | 4 周 |
| **2.2** 小额自动执行 | 低风险操作无需人工介入 | 2.1 稳定运行，采纳率 ≥ 70% | 3 周 |
| **2.3** 分级授权 | 按信任度扩大自动决策范围 | 2.2 稳定 4 周 | 2 周 |

**整体原则：** 每个子阶段必须评测通过后才能进入下一阶段，不跳级。数据是基础，准确率是门槛，安全是红线。
