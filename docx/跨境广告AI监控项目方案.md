# 跨境广告 AI 监控项目方案

## 一、项目背景

团队现状：3 名投手分别负责不同品类，手动登录 Facebook / Google / TikTok 后台查看数据，无统一看板，靠人工盯，容易漏看。

核心痛点：
- 高消耗低 ROI 广告组未及时关停，持续亏钱
- 高 ROI 优质广告组未及时加预算，错过放量窗口
- 低点击率素材长期跑，浪费预算，未及时通知设计师换素材

---

## 二、整体路线

```
第一阶段（立刻落地）  →  纯规则预警，解决漏看问题
第二阶段（数据成熟后）→  AI 轻决策，辅助自动调价
```

---

## 三、第一阶段：纯规则预警系统

### 3.1 目标

零 AI 决策，零误操作风险。用规则引擎替代人工盯盘，有异常自动发企微通知。

### 3.2 预警规则（每位投手可按品类独立配置阈值）

| 规则名 | 触发条件 | 通知对象 | 建议动作 |
|--------|----------|----------|----------|
| 高消耗低 ROAS | 当日消耗 ≥ X 元 且 ROAS ≤ 最低盈利红线 | 对应投手 | 提醒关停该广告组 |
| 高 ROAS 可放量 | ROAS ≥ 增量线阈值 且 消耗未达预算上限 | 对应投手 | 提醒加预算 |
| 低点击率素材 | 近 N 天 CTR ≤ 点击率红线 且 展示量 ≥ M | 投手 + 设计师 | 提醒更换素材 |
| 预算即将耗尽 | 剩余预算 ≤ 当日日预算的 10% | 对应投手 | 提醒决策是否续预算 |

> **判断粒度说明：**
> - 消耗 / ROAS / 预算规则 → **Ad Set（广告组）**层级：Meta 用 AdSet ID，Google / TikTok 用 AdGroup ID。一个 Campaign 下有几十条 Ad Set，每条独立统计消耗和出单，告警必须精确到这一层。
> - 点击率规则 → **Ad（广告创意）**层级：Ad ID 对应单条图片/视频素材，专门判断素材质量，触发后通知设计师。
> - Campaign 绑定只作为"品类归属"的容器，系统从已绑定 Campaign 下遍历所有 Ad Set 和 Ad 进行判断。
>
> 阈值均支持按品类独立配置，由投手在 Web 界面自助设定。

### 3.3 系统架构

```
          ┌──────────────────────┐
          │  Web 配置界面（投手）  │  ← 品类/阈值/人员 userid 管理
          └──────────┬───────────┘
                     │ 读写配置
┌────────────────────▼────────────────────┐
│              数据存储（DB）               │  ← 配置 + 历史数据
└──────┬─────────────────────┬────────────┘
       │                     │
┌──────▼──────┐     ┌────────▼────────┐
│  定时调度器  │     │   规则引擎       │  ← 按品类加载阈值
│  （每小时）  │────▶│                 │
└─────────────┘     └────────┬────────┘
       │                     │ 触发预警
┌──────▼──────┐     ┌────────▼──────────────┐
│  数据拉取模块│     │  企微群 Webhook 通知    │
│ Facebook API│     │  @对应投手 / @设计师    │
│ Google API  │     └───────────────────────┘
│ TikTok API  │
└─────────────┘
```

### 3.4 企微通知说明

所有预警统一推送到**一个企微群**（Webhook 地址全局配置一次），通过 `mentioned_list` 字段 @指定成员，无需多个 Webhook。

| 预警类型 | @通知对象 |
|----------|-----------|
| 高消耗低 ROI（关停提醒） | @该品类投手 |
| 高 ROI 可放量（加预算提醒） | @该品类投手 |
| 预算即将耗尽 | @该品类投手 |
| 低点击率素材 | @该品类投手 + @对接设计师 |

**通知样例——投手预警（高消耗低 ROI）：**
```
[广告预警] 高消耗低 ROI
@张三（投手）
平台：Facebook | 品类：女装
广告组：女装-夏季连衣裙-A组
今日消耗：¥2,340 | ROI：0.8（红线：1.2）
建议：立即暂停该广告组
```

**通知样例——素材预警（低点击率，@投手 + @设计师）：**
```
[素材预警] 点击率过低
@张三（投手）@李四（设计师）
平台：TikTok | 品类：女装
素材 ID：video_20240601_003
近 7 天 CTR：0.6%（红线：1.2%）| 展示量：45,000
建议：更换该素材
```

### 3.5 Web 配置界面

投手通过浏览器访问配置页面，无需编辑任何文件。

**全局配置（管理员设置一次）：**

| 配置项 | 说明 |
|--------|------|
| 企微群 Webhook URL | 所有预警推送到同一个群 |

**品类配置（每个品类独立一行，投手自助维护）：**

| 配置项 | 说明 |
|--------|------|
| 品类名称 | 如：女装、3C数码、家居 |
| 负责投手企微 userid | 用于 @mention，如 `zhangsan` |
| 对接设计师企微 userid | 用于素材预警 @mention |
| 最低盈利 ROI 红线 | 低于此值且消耗达标 → 关停提醒 |
| 高 ROI 增量线 | 高于此值 → 加预算提醒 |
| 触发 ROI 预警最低消耗（元） | 避免小额消耗误报 |
| CTR 红线 | 低于此值 → 通知换素材 |
| CTR 统计周期（天） | 默认 7 天 |
| CTR 最低展示量 | 展示量不足不触发预警 |

**广告活动绑定（品类创建后，投手自助操作）：**

投手将自己在平台上负责的 Campaign ID 绑定到对应品类，系统只拉取已绑定的 Campaign 数据。

| 操作 | 说明 |
|------|------|
| 选择平台 | facebook / google / tiktok |
| 输入 Campaign ID | 从平台后台复制粘贴，支持一个品类绑定多条 |
| 新建 Campaign 时 | 投手在平台上创建后，补录到 Web 界面即可 |

界面交互：列表展示所有品类 → 点击编辑 → 填写基本信息和阈值 → 进入"绑定广告活动"子页面添加 Campaign ID → 保存后实时生效，无需重启服务。

**数据展示与刷新策略：**

| 机制 | 说明 |
|------|------|
| 默认展示 | 页面加载时从 DB 读取最新快照（上次调度器拉取的结果） |
| 自动刷新 | 调度器每小时自动拉取全部已绑定 Campaign，更新快照并执行规则判断 |
| 手动刷新 | 品类详情页提供「立即刷新」按钮，仅拉取**当前品类**已绑定 Campaign 下的所有 Ad Set 和 Ad 数据，更新快照后同步执行规则判断（触发条件时即时发出企微告警）；新绑定的 Campaign 手动刷新后即可在页面看到其下广告组数据，验证绑定是否正确 |
| 冷却限制 | 手动刷新按钮 5 分钟内只能触发一次，防止频繁调用打满平台 API 配额 |

> 三平台报告数据本身有延迟（Facebook/Google 约 1~3 小时，TikTok 约 1 小时），手动刷新拿到的是平台当前最新可用数据，非真正实时。

### 3.6 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 语言 | Python 3.10+ | 三平台均有官方 Python SDK |
| Web 框架 | FastAPI + Jinja2 | 配置界面 + API，轻量易维护 |
| 调度 | APScheduler | 内嵌进程，无需独立部署 |
| 存储 | SQLite → 后期 MySQL | 第一阶段 SQLite 够用，数据量大后迁移 |
| 通知 | 企微群 Webhook | 单一群地址，用 mentioned_list @指定成员 |
| 部署 | 公司内网服务器 | 内网访问，Web 界面仅内部可达 |

### 3.7 第一阶段交付物

- [ ] 三平台 API 数据拉取模块（Ad Set + Ad 两层）
- [ ] 规则引擎（按品类加载阈值，Ad Set 判断 ROAS/消耗/预算，Ad 判断 CTR）
- [ ] 企微群 Webhook 通知（@投手 / @投手+设计师 分场景）
- [ ] Web 配置界面（品类管理、阈值设置、人员 userid 录入、Campaign 绑定子页面）
- [ ] 品类数据展示页 + 手动刷新按钮（5 分钟冷却）
- [ ] Ad Set / Ad 两层数据快照入库（供第二阶段训练）
- [ ] 定时调度（每小时跑一次）

**数据库表（SQLite）：**

| 表名 | 用途 |
|------|------|
| `categories` | 品类配置：名称、负责投手 userid、设计师 userid、各项阈值 |
| `campaign_bindings` | Campaign 绑定：campaign_id、platform、关联 category_id |
| `global_config` | 全局配置：企微群 Webhook 地址 |
| `adset_snapshots` | Ad Set 层级快照：adset_id、campaign_id、platform、category_id、spend、roas、remaining_budget、snapshot_at |
| `ad_snapshots` | Ad 层级快照（素材）：ad_id、adset_id、platform、category_id、ctr、impressions、snapshot_at |
| `alert_records` | 预警去重记录，字段：id、entity_id（AdSet ID 或 Ad ID）、entity_type（adset / ad）、platform、category_id、rule_type（high_spend_low_roas / high_roas_scale / budget_low / low_ctr）、triggered_at；查询时用 entity_id + rule_type + triggered_at 判断 24 小时内是否已发过 |

## 四、第二阶段：AI 轻决策（数据积累后迭代）

> 前提：第一阶段至少积累 4~8 周数据，并验证规则预警准确率 ≥ 85%。

### 4.1 目标

在人工可审核的前提下，AI 给出调价/关停建议，投手一键确认执行，减少判断时间。

### 4.2 迭代路径

```
阶段 2.1  AI 给建议，人工一键执行（企微卡片 → 确认按钮）
    ↓     验证 AI 建议准确率，跑评测
阶段 2.2  小额低风险操作（如 ±10% 调价）允许 AI 自动执行，超阈值仍需人工确认
    ↓     持续评估 ROI 影响
阶段 2.3  开放更多自动决策范围（按投手信任度分级授权）
```

### 4.3 AI 决策输入特征（由第一阶段数据积累）

- 广告组历史 ROI 趋势（近 7/14/30 天）
- 消耗加速度（今日 vs 昨日 vs 近 7 天均值）
- 素材表现衰减曲线（CTR 随投放天数变化）
- 同品类同平台基准（自建 baseline，非第三方）
- 历史人工操作记录（投手实际关停/加钱的时机）

### 4.4 评测体系

上线 AI 建议前，必须完成：
- 回测：用历史数据跑规则，对比投手实际操作，准确率需 ≥ 85%
- 仿真期：AI 建议与人工决策并行 2 周，记录分歧
- 上线标准：AI 建议采纳率 ≥ 70%，且采纳后 ROI 不低于人工操作均值

## 五、项目排期（第一阶段）

| 周次 | 任务 |
|------|------|
| 第 1 周 | API 接入（Facebook + Google + TikTok），数据拉取验证 |

| 第 4 周 | 内测（3 位投手试用），收集反馈，调整阈值，稳定运行 |

## 六、风险与注意事项

1. **API 认证方式待定（待接入）**：三平台认证机制不同（Facebook 用 OAuth 2.0 长期 Token、Google 用 Service Account、TikTok 用 App Access Token），Token 刷新和过期处理逻辑尚未设计，为第一阶段开发前的必要前置工作。
2. **API 配额限制**：三平台 API 均有调用频率限制，需合理设置拉取间隔（建议每小时一次，非实时）。
3. **Token 安全**：各平台 API Key 存储在环境变量或加密配置中，不进代码仓库。
4. **第一阶段只通知、不自动操作**：所有广告操作仍由投手手动完成，系统不调用任何写接口，零误操作风险。
5. **企微 Webhook 防刷**：同一广告组同一规则，24 小时内只通知一次，避免重复打扰。
6. **ROAS 口径**：三平台均直接取各自后台的 ROAS 数值，投手按品类设置 ROAS 阈值即可，无需换算。

---

## 七、方案总结

- **第一阶段核心价值**：用自动化规则巡检替代人工盯盘，解决漏看问题，零风险落地，1 个月内可上线。
- **第二阶段核心价值**：基于真实数据训练 AI 判断模型，逐步从"提醒"升级为"建议+一键执行"，再到"小范围自动决策"，可控迭代。
- **整体原则**：先省人力、后提效率；先人工兜底、后 AI 接管；数据是基础，准确率是门槛。
